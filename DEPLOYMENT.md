# Deployment Guide

This document describes how to deploy Funding.Wiki to a production server.
For local development setup, see the [project README.md](README.md).

## Architecture

```
Internet → public-ip-server (Nginx + SSL/TLS, public IP)
                ↓ Tinc VPN (<VPN_NETWORK>)
           app-server (Gunicorn + Whitenoise + PostgreSQL/PostGIS + Redis)
```

* **public-ip-server**: public-facing server. Handles SSL termination and proxies all requests to the application server via Tinc VPN.
* **app-server**: application server. Runs the Django app (Gunicorn), database (PostgreSQL/PostGIS), cache (Redis), and serves static files (Whitenoise).

> NOTE: Libretranslate currently is in pulic-ip-server, but we'll be move to improve speed in translations.

## Prerequisites

### On public-ip-server

* Docker and Docker Compose
* Tinc VPN configured, with app-server reachable at `<VPN_APP_SERVER_IP>`
* Nginx and Certbot running via Docker (`~/nginx-proxy/`)
* SSL certificate obtained for `funding.wiki`

### On app-server

* Docker and Docker Compose
* Tinc VPN configured
* The project running inside the devcontainer at `/workspace`
* Node.js and npm available inside the container

## First-time setup

### 1. Clone the repository (on app-server, inside the .devcontainer)

```bash
git clone https://github.com/freedomsponsors/fundingwiki /workspace
cd /workspace
```

### 2. Run the setup script

```bash
.devcontainer/scripts/setup.sh
```

This will:

* Wait for PostgreSQL to be ready
* Run database migrations
* Load initial data (as language tables and others)
* Create a superuser (you will be prompted for credentials)
* Install frontend dependencies

### 3. Configure environment variables

Create a `.env` file at the project root (see `.env.example` if available):

```
DATABASE_NAME=djangology
DATABASE_USER=djangology
DATABASE_PASS=your_password
DATABASE_HOST=db
DATABASE_PORT=5432
```

### 4. Run the first deploy

```bash
./deploy.sh
```

---

## Deploying updates

Every time you want to deploy a new version, run from inside the devcontainer on app-server:

```bash
./deploy.sh
```

The script will:

1. Pull the latest code from `master`
2. Install any new Python dependencies
3. Run database migrations
4. Build the Vue frontend (`npm run build`)
5. Copy the generated `index.html` to `templates/vue/index.html`
6. Reload Gunicorn (or start it if not running)

---

## What the deploy script does NOT do

* It does not restart the database or Redis
* It does not update Nginx configuration on the public server
* It does not renew SSL certificates (Certbot handles this automatically)

---

## Frontend build notes

Vite is configured to output production assets directly to `statfiles/static/vue/`. The `VITE_BASE_PATH` variable is set to `/static/vue/` via `frontend/.env.production`, so no manual configuration is needed.

Do **not** commit `templates/vue/index.html` — it is generated by the build step and listed in `.gitignore`.

Do **not** commit `statfiles/static/vue/` — same reason.

---

## Static files

Static files are served by Whitenoise, which is integrated into Django's middleware stack. No separate Nginx configuration is needed for `/static/`.

## Switching Nginx targets (public-ip-server)

Two Nginx configurations are maintained on the public server:

| File               | Target                                                  | Use                                                                     |
| ------------------ | ------------------------------------------------------- |-------------------------------------------------------------------------|
| `nginx.conf.app`   | application server via Tinc VPN (`<VPN_APP_SERVER_IP>`) | current                                                                 |
| `nginx.conf.local` | local devcontainer running on the public server itself  | ready for future if public-ip-server is more powerful than a VPN server |

To switch between them:

```bash
# Point to app-server
cp ~/nginx-proxy/nginx/nginx.conf.app ~/nginx-proxy/nginx/nginx.conf
docker compose -f ~/nginx-proxy/docker-compose.yml restart nginx

# Point to local devcontainer
cp ~/nginx-proxy/nginx/nginx.conf.local ~/nginx-proxy/nginx/nginx.conf
docker compose -f ~/nginx-proxy/docker-compose.yml restart nginx
```

## SSL certificate renewal

SSL certificates are renewed automatically by the Certbot container on the public server. No manual intervention is needed.

To force a manual renewal:

```bash
cd ~/nginx-proxy
docker compose run --rm \
    --entrypoint "certbot renew --webroot -w /var/www/certbot" \
    certbot
docker compose restart nginx
```

## Backups

Database backups are currently not automated.

Manual backup:

```bash
pg_dump -h db -U djangology djangology > backup.sql
```

Restore:

```bash
psql -h db -U djangology djangology < backup.sql
```

## Troubleshooting

**Gunicorn is not running after a server restart**

If the devcontainer restarts, Gunicorn may need to be started manually:

```bash
gunicorn config.wsgi:application --bind 0.0.0.0:8000 --workers 3 --daemon
```

A proper production setup using `docker-compose.prod.yml` with automatic restart is planned.

---

**Static files return 404**

Make sure the frontend has been built and `templates/vue/index.html` has been updated:

```bash
cd frontend && npm run build && cd ..
cp statfiles/static/vue/index.html templates/vue/index.html
```

---

**Database connection errors**

Check that the PostgreSQL container is running:

```bash
docker compose ps
```
